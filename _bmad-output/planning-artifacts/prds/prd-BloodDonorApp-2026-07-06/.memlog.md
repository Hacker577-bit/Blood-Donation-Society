- (decision) Product: Blood Donor Availability Matcher. Replaces disorganized WhatsApp-group scrambling during blood emergencies with structured donor search.
- (decision) Core donor registration fields: name, phone, blood type, location (browser geolocation lat/long), last donation date.
- (decision) Eligibility rule: 90+ days since last donation.
- (decision) Search: filter by blood type, sort by distance (Haversine, no paid API), radius auto-widens 15km -> 30km -> 50km if no match, then clear empty state (never fail silently).
- (decision) Donor self-service: update last donation date, delete own registration.
- (decision) Rate limiting: basic per-IP only for MVP.
- (decision) Out of scope MVP: in-app chat, hospital integration, donation verification, push notifications, paid mapping API.
- (decision) Tech stack: Python Flask, PostgreSQL. Frontend: polished/exquisite visual design, explicitly NOT 3D (addendum note).
- (decision) Success criteria: donor registers in under 1 minute; searcher finds eligible nearby donor in under 30 seconds.
- (decision) Stakes: internal tool for an org. Working mode: Fast path.
- (decision) Contact model pivoted from 'reveal donor phone to searcher' to privacy-preserving request-broadcast: searcher submits their own contact number with the request (does not see donor list/numbers directly).
- (decision) System automatically notifies matching eligible donors nearby via WhatsApp message that someone needs their blood type; donor decides whether to reach out to searcher's saved contact number.
- (decision) WhatsApp notification requires WhatsApp Business API (Meta) integration - a paid/approval-gated external dependency, accepted by user despite general 'no paid API' MVP preference (that preference was scoped to mapping specifically).
- (decision) Donor auth for self-update/delete: OTP (one-time password) sent to donor's phone to prove ownership.
- (assumption) Org name, exact user-scale target, and formal compliance/data-protection constraints not specified by user - to be tagged [ASSUMPTION] in draft PRD for review.
- (event) Draft PRD (prd.md) and addendum.md written for fast-path review.
- (decision) MAJOR REVISION: reverted privacy-broadcast model. Searcher registers name+phone before searching; sees FULL donor list (name, phone, area) directly - direct reveal reinstated.
- (decision) Location model changed: dropped GPS/geolocation and Haversine entirely. Replaced with predefined dropdown of Lahore-only localities (Johar Town, DHA, Gulberg, Model Town, Bahria Town, Cantt, Iqbal Town, Garden Town, Wapda Town, Faisal Town). Exact-match on area. Donors can select multiple areas they'll travel to. DECIDED - not open for change.
- (decision) No-match handling: suggest expanding to nearby areas, then clear empty state if still no match. Ineligible donors never shown as fallback. DECIDED.
- (decision) Notification channel changed again: dropped WhatsApp (both Business API and wa.me link ideas) entirely. Final: SMS via Twilio free trial + email via SendGrid free tier, both sent automatically when search is submitted. DECIDED.
- (assumption) Risk flagged to user: Twilio free trial can only SMS numbers manually verified in the Twilio console - will not reach real donors' numbers until account is upgraded to paid. Logged as NOTE FOR PM, not blocking per user's 'accept'.
- (assumption) Area-adjacency ('nearby areas') list for no-match expansion not specified by stakeholder - which Lahore localities count as neighboring which. Flagged as open question.
- (event) PRD and addendum rewritten to reflect area-based matching, direct donor-list reveal, and Twilio SMS + SendGrid email notification. User said accept.
- (event) Reviewer gate (rubric walker) run. Verdict: decision-ready with fixable gaps. Fixed: Assumptions Index roundtrip, FR-6/FR-7 UJ tagging. Open: searcher abuse/cost vector (High), missing PII/data-handling NFR (Medium).
- (decision) Searcher abuse/cost vector resolved: Searchers must OTP-verify their phone before a search returns results or fires donor notifications (mirrors donor OTP). Resolves prior Open Question 4. New open question: does searcher verification persist per session or re-fire every search (time-budget vs friction tradeoff).
