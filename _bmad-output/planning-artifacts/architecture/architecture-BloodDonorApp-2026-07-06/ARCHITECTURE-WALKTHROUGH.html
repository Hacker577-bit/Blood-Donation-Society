Design paradigm

Layered monolith, one dependency direction

One Next.js App Router app. No separate API server. Domain logic never touches a concrete adapter directly — only through ports — so eligibility and matching are testable without a database, SMS, or email in the loop.

Structural seed

Core entities

AD-1

Searcher has no persisted identity

Binds
FR-5, FR-8
Prevents
A reusable Searcher table gaining its own lifecycle/auth/dedup rules the PRD never specified.
Rule
Name and phone are inlined fields on the Search record only. No Searcher table exists.
AD-2 Adopted

OTP challenges live only in Redis

Binds
FR-2, FR-5, FR-9 — all three OTP flows
Prevents
Registration, search, and self-service OTP each inventing its own expiry/cleanup strategy.
Rule
Every OTP challenge (phone + purpose → codeHash, expiresAt, attempts) is a Redis key with a TTL (~5 min). No OTP table in the relational schema. One shared otp service.
AD-3 Tightened at review

One shared rate-limit utility

Binds
FR-12 — registration, search, OTP endpoints
Prevents
Per-endpoint counters drifting on window/key/response shape; two builders disagreeing on what "the client IP" means behind Vercel's proxy.
Rule
One Redis sliding-window utility keyed by ip + endpoint, where ip is always the platform-trusted resolved client IP — never a raw client-supplied x-forwarded-for entry. Shared 429 response shape.
AD-4 Tightened at review

Post-OTP session: short-lived, server-enforced token

Resolves PRD Open Question 6.
Binds
FR-5, FR-9, FR-10, FR-11 — not FR-1/FR-2 (registration)
Prevents
Re-verifying OTP every action vs. a full persistent login; a client reusing a token past its intended single/bounded use.
Rule
JWT, ~15 min TTL, scoped to one phone/Donor id, unique jti. Server tracks remaining uses in Redis and rejects once spent — a stateless "trust the TTL" build does not satisfy this. Searcher budget: 2 (search + one area-expansion re-search). Self-service: 1 use (update or delete). Registration issues no token at all — OTP there activates the record directly.
AD-5

Eligibility is derived, never stored

Binds
FR-3, FR-4
Prevents
A later "perf" pass caching isEligible and letting it drift stale relative to lastDonationDate.
Rule
isEligible = (today - lastDonationDate) >= 90d OR lastDonationDate IS NULL, computed at query time in the domain layer. Never persisted.
AD-6 Tightened at review

Notification never blocks the Searcher's response

Binds
FR-8
Prevents
Inline Twilio/SendGrid calls eating into the 30-second SM-2 budget; a silently-dropped failure log if the platform tears the function down before a detached promise resolves.
Rule
Match response returns immediately; dispatch runs in Next.js' post-response after() hook. The after() callback awaits the full dispatch-and-log sequence before returning — never fire-and-forget inside after().
AD-7

Area adjacency is static code, not a resource

Binds
FR-6
Prevents
One builder wiring a DB-backed, admin-editable adjacency map while another hardcodes it — two sources of truth.
Rule
One static, versioned TypeScript module keyed by the Area enum, imported by the domain matching service.
Open
The concrete adjacency values are still a PM/product decision (PRD Open Question 1) — this AD only fixes that it's code, not data.
Consistency conventions

Defaults that prevent drift

ConcernConvention
NamingDomain services/ports under lib/domain/; adapters under lib/infra/; one file per concern.
Data & formatsIDs: Prisma cuid(). Phone: E.164 everywhere. Dates: ISO 8601 date-only. Area/BloodType: fixed enums. Errors: { error: { code, message } }.
State & cross-cuttingAll writes through lib/infra/repositories/* — no direct Prisma calls from app/. Zod validation at the Presentation boundary. Secrets as env vars only.

Intentionally pushed down

  • Exact per-IP rate-limit thresholds (Open Question 3) — tunable config on top of AD-3.

  • SendGrid free-tier volume vs. expected traffic (Open Question 4) — depends on unknown org scale.
  • Formal PII/data-protection compliance review (Open Question 5) — legal scope.
  • Observability/monitoring stack — revisit once hosting + real traffic exist.
  • Testing strategy detail (unit/integration split, CI) — implementation planning; the port boundaries make it possible, not prescribed.
