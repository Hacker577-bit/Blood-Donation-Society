- (decision) [ASSUMPTION] Paradigm: layered monolith with service-boundary ports, single Next.js App Router app (no separate API server). Presentation = app/ routes + Server Actions; Domain = pure services (eligibility, matching, OTP issuance rules); Infrastructure = adapters implementing thin ports for Prisma, Twilio, SendGrid, Redis. Binds: all. Prevents: business rules (eligibility calc, matching) becoming untestable because they are inlined in route handlers next to Twilio/SendGrid calls.
- (decision) [ASSUMPTION] Searcher is NOT a persisted reusable entity -- name+phone are inlined fields on the Search record only, matching PRD 6.2 'no persistent Searcher accounts'.
- (decision) [ASSUMPTION] OTP challenges (Donor reg, Searcher verify, self-service entry) live in Redis only, keyed by phone+purpose, with TTL-based expiry (~5min per PRD assumption) -- never a Postgres table. Prevents divergent expiry/cleanup logic (cron job vs TTL) between two builders.
- (decision) [ASSUMPTION] Rate limiting (FR-12) is enforced by ONE shared Redis sliding-window utility keyed by IP+endpoint, invoked from a single middleware layer -- not reimplemented per route.
- (decision) [ASSUMPTION] Resolves PRD Open Question 6: after OTP verify, a short-lived signed token (JWT, ~15min TTL, scoped to the verified phone/donorId) is issued and used as bearer auth for that session's subsequent calls (Searcher: submit search + one area-expansion re-search; Donor self-service: update/delete). A new visit always requires fresh OTP -- this is NOT a persistent login.
- (decision) [ASSUMPTION] Eligibility is never stored as a flag/column -- always derived at query time from lastDonationDate (already required by FR-3; elevated to AD so a builder doesn't add a cached isEligible column for perf and let it drift stale).
- (decision) [ASSUMPTION] Donor Notification dispatch (FR-8) runs via Next.js' post-response after() hook -- never inline before the Searcher's match list is returned. Prevents a builder from accidentally blocking the Searcher response on Twilio/SendGrid latency.
- (decision) [ASSUMPTION] Area adjacency (PRD Open Question 1) is stored as a static, versioned TypeScript config module keyed by the Area enum -- not a DB table or admin-editable resource. The actual adjacency values remain a product decision (Open Question 1 stays open for PM), but the architectural shape (static code, not data) is fixed so two builders don't diverge into one building a DB-backed adjacency admin UI.
- (convention) Phone numbers normalized and stored in E.164 format everywhere (uniqueness key for Donor, Twilio compatibility).
- (convention) Area stored as a fixed 10-value enum; BloodType stored as a fixed 8-value ABO/Rh enum; lastDonationDate stored as a date-only ISO 8601 value (no time-of-day).
- (convention) IDs are Prisma-generated cuid() strings throughout.
- (convention) Server Action / Route Handler error shape is a consistent envelope: { error: { code, message } }; validation via Zod schemas colocated with each domain input.
- (constraint) Twilio free-trial accounts cannot reach unverified real numbers -- functional launch blocker per PRD Cost section, owned by ops/PM before go-live, not an architecture concern.
- (constraint) SendGrid free tier daily send cap should be checked against expected search volume before launch (PRD Open Question 4 -- org scale unknown).
