# Story 2.2: Searcher Submits Search, Views Matches, and Matched Donors Are Notified

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a searcher,
I want to submit a blood type and one Area and immediately see every eligible matching donor's name, phone, and area, while those donors are notified,
so that I can start calling directly without delay, and donors who might reach out first are alerted right away.

## Acceptance Criteria

1. **Given** I have a valid Searcher session token (Story 2.1) with remaining budget, **when** I select a blood type and one Area (via the single-select Area chip, UX-DR7) and tap Search, **then** the matching domain service returns every Donor who is eligible (per AD-5), shares the requested blood type, and has the searched Area among their selected Areas — ineligible Donors are never included, regardless of blood type. [Source: epics.md#Story-2.2, FR-4, AD-5]
2. **And** this story records the submitted search in the `Search` Prisma table (searcherName, searcherPhone E.164, bloodType, area, createdAt) — the first story that writes to it. [Source: epics.md#Story-2.2, Data model]
3. **And** I see skeleton list rows while the search runs, not a spinner-only blank screen, and the full result (name, phone, Area per Match) is returned directly, not anonymized. [Source: epics.md#Story-2.2, UX-DR5]
4. **And** each Donor match card shows name and phone in `body-large` type with Area as a `meta` tag, and the phone number is a tap-to-call `tel:` link on mobile / copy-to-clipboard on desktop. [Source: epics.md#Story-2.2, UX-DR2/UX-DR7]
5. **And** once Matches are computed, my response returns immediately without waiting for SMS/email delivery — dispatch runs via the framework's post-response `after()` hook, which itself awaits the full dispatch-and-log sequence before returning; delivery failures are logged, never surfaced to me. [Source: epics.md#Story-2.2, AD-6, FR-8]
6. **And** every Match receives both an SMS (Twilio) and an email (SendGrid, if they have one on file — SMS-only otherwise) containing my name, phone number, the blood type needed, and the Area. [Source: epics.md#Story-2.2, FR-8]
7. **And** this action consumes one unit of my session token's 2-use budget. [Source: epics.md#Story-2.2, AD-4]

- [x] Task 4: Build the `submitSearch` Server Action (AC: #1, #2, #5, #6, #7)

  - [x] Create `app/actions/submitSearch.ts` (new):
    - Takes `{ sessionToken: string; searcherName: string; bloodType: string; area: string }`.
    - Rate-limit first, before touching the session token: `checkRateLimit({ ip, endpoint: "submitSearch" }, redisRateLimitStore, config)` using **new** dedicated env vars `RATE_LIMIT_SEARCH_MAX`/`RATE_LIMIT_SEARCH_WINDOW_SECONDS` (default `5`/`60`, same defaulting pattern as every other rate-limited action). This deviates from Story 2.1's choice to reuse the OTP env-var pair — justified because `submitSearch` is not an OTP-category action (it's the actual cost-triggering search/notification gate that NFR-4/NFR-7 call out), so it gets its own dedicated pair per Story 1.2's original one-pair-per-endpoint precedent. On rejection: `{ error: { code: "RATE_LIMITED", message: "Too many attempts. Please try again shortly." } }` (identical copy to every other rate-limited action).
    - Validate `{ searcherName, bloodType, area }` via `submitSearchSchema.safeParse` → `{ error: { code: "VALIDATION_ERROR", message: "Please fix the highlighted fields and try again.", fieldErrors } }` on failure (same shape/copy as `registerDonor.ts`).
    - Verify the session token: `verifySessionToken(sessionToken, joseTokenSigner)`. If `null` → `{ error: { code: "SESSION_INVALID", message: "Your session has expired. Please verify your phone again." } }`.
    - Consume one unit of budget: `consumeSessionUse(jti, redisSessionBudgetStore)`. If `allowed` is `false` → `{ error: { code: "SESSION_EXHAUSTED", message: "This search session has been used up. Please verify your phone again." } }`. This decrement is what satisfies AC #7 — it must happen exactly once per successful `submitSearch` call, before any DB write, so a search is never both recorded and left un-decremented (or vice versa) on a retry.
    - Run the match: `findMatches({ bloodType, area }, donorRepository, new Date())` (the extended `donorRepository.ts` from Task 2 satisfies the `DonorMatchLookup` port structurally — no adapter object needed beyond the module itself, same as how `redisOtpStore`/`redisRateLimitStore` are passed directly). **Implementation note:** `matching.ts`'s `DonorMatch` also carries `email` (not just `name`/`phone`/`area`) — needed by `notifyMatches` below to decide SMS-only vs. SMS+email; strip it back out of the client-facing response (`matches.map(({ name, phone, area }) => ({ name, phone, area }))`), since the Searcher never sees a donor's email.
    - Record the search: `createSearch({ searcherName, searcherPhone: verifiedToken.subject, bloodType, area })` via `searchRepository.ts` — **`searcherPhone` always comes from the verified token's `subject`, never from client input** (see validation schema note above).
    - Build the response `{ matches: matches.map(({ name, phone, area }) => ({ name, phone, area })) }` and prepare to return it **before** scheduling notification dispatch.
    - Schedule notification dispatch via `after()` (import `{ after } from "next/server"`): `after(async () => { try { await notifyMatches(matches, { searcherName, searcherPhone: verifiedToken.subject, bloodType: BLOOD_TYPE_LABELS[bloodType], area: AREA_LABELS[area] }, twilioNotificationSender, sendgridEmailNotifier); } catch (err) { console.error("submitSearch: notification dispatch failed", err); } })` — pass the full `matches` (including `email`) to `notifyMatches`, and resolve `bloodType`/`area` to their human-readable labels here (via `lib/presentation/labels.ts`) since `notify.ts` itself takes pre-resolved display strings, not raw enums. The `after()` callback itself must `await` the full `notifyMatches` call to completion (AD-6 — "never fire-and-forget *inside* `after()`"); the outer `try/catch` is a last-resort guard since `notifyMatches` (Task 3) already catches per-match, but the callback itself must never throw and crash the post-response phase.
    - Return the matches response. The action must return as soon as matches + the `Search` record are ready — `after()`'s scheduling is what keeps the notification dispatch from delaying this response (Next.js 16's `after()` is stable, not experimental, as of this project's Next 16.2.x — see Latest Tech Notes).

## Dev Notes

### Architecture Compliance
- Layered monolith, one dependency direction: Presentation (`app/`) → Domain (`lib/domain/`) → Infrastructure (`lib/infra/`), through port interfaces. `lib/domain/matching.ts` and `lib/domain/notify.ts` must not import Prisma, `twilio`, or `@sendgrid/mail` directly — same discipline as `otp.ts`/`rate-limit.ts`/`session.ts`. [Source: ARCHITECTURE-SPINE.md, Design Paradigm]
- AD-1 (binds FR-5, FR-8): `searcherName`/`searcherPhone` are inlined fields on the `Search` record only — this story is the one that actually writes them; still no `Searcher` table. `searcherPhone` must be sourced from the verified session token's `subject`, never from client-submitted form data (Task 4). [Source: ARCHITECTURE-SPINE.md, AD-1]
- AD-3 (binds FR-12): `submitSearch` must go through the shared `checkRateLimit` utility with its own `endpoint: "submitSearch"` bucket — do not add a second rate-limiting mechanism. [Source: ARCHITECTURE-SPINE.md, AD-3]
- AD-4: this story is the **first consumer** of `verifySessionToken`/`consumeSessionUse` from `lib/domain/session.ts` (built in Story 2.1 but not yet exercised by any AC there) — reuse both verbatim, do not modify `session.ts`. A stateless "trust the TTL" check is insufficient; the Redis-tracked budget must actually be decremented (Story 2.1's `consumeSessionUse` already implements this correctly). [Source: ARCHITECTURE-SPINE.md, AD-4; Story 2.1 Dev Notes]
- AD-5 (binds FR-3, FR-4): eligibility is computed at query time in the domain layer via `computeEligibility` (`lib/domain/eligibility.ts`, built in Story 1.4) — `matching.ts` must call this existing function, not reimplement the 90-day rule. [Source: ARCHITECTURE-SPINE.md, AD-5]
- AD-6 (binds FR-8): notification dispatch runs via `after()` from `next/server`, which must itself `await` the full dispatch-and-log sequence before returning — never a bare fire-and-forget call inside `after()`. Failures are logged (via `console.error`, matching `registerDonor.ts`'s existing error-logging style), never surfaced to the Searcher. [Source: ARCHITECTURE-SPINE.md, AD-6]
- Capability → Architecture Map: "Search & Area Expansion (FR-5, FR-6, FR-7) | `app/actions/submitSearch`, `lib/domain/matching.ts` | AD-1, AD-3, AD-4, AD-7" and "Donor Notification (FR-8) | `lib/domain/notify.ts`, `lib/infra/twilio.ts`, `lib/infra/sendgrid.ts`" — this story builds exactly these named files. [Source: ARCHITECTURE-SPINE.md, Capability → Architecture Map]
- Error shape convention (`{ error: { code, message } }`) applies to `submitSearch`, matching every existing action. [Source: ARCHITECTURE-SPINE.md, Consistency Conventions]
- Domain/Infra separation: "all writes go through Prisma inside `lib/infra/repositories/*` — no direct Prisma calls from `app/`" — `submitSearch` calls `donorRepository`/`searchRepository`, never `prisma` directly. [Source: ARCHITECTURE-SPINE.md, Consistency Conventions]

### Latest Tech Notes (verified 2026-07-11)
- **Next.js `after()`**: stable (not experimental) as of Next.js 16 — this project is on `16.2.10`, well past the graduation point. Usable directly in Server Actions/Route Handlers via `import { after } from "next/server"`; schedules work to run once the response has been sent, without blocking it. The callback should `await` any async work it starts — `after()` keeps the serverless function alive until the callback's promise settles, but does not itself await anything inside a fire-and-forget call you make without `await`. [Source: nextjs.org/docs/app/api-reference/functions/after, verified via web search 2026-07-11]
- **`@sendgrid/mail`**: current stable is `8.1.6` on npm (verified 2026-07-11), the official Twilio SendGrid-maintained Node.js mail client. `sgMail.setApiKey(...)` once at module scope, then `sgMail.send({ to, from, subject, text })` — `from` must be a SendGrid-verified sender identity/domain in production (not enforced in this story's tests, which mock the SDK entirely). [Source: npmjs.com/package/@sendgrid/mail, verified via web search 2026-07-11]
